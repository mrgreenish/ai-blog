# Article Writing

Write long-form content that sounds like a real person, not generic AI output.

## When to Activate

- drafting blog posts, essays, launch posts, guides, tutorials, or newsletter issues
- turning notes, transcripts, or research into polished articles
- matching an existing founder, operator, or brand voice from examples
- tightening structure, pacing, and evidence in already-written long-form copy

## Default Voice: Curious Explainer

The default voice is modeled on makingsoftware.com (Dan Hollick) — a design engineer who explains complex technical concepts like a smart friend at a whiteboard. Apply these traits unless the user supplies a different voice reference.

### Tone and Rhythm

- Warm, conversational, genuinely curious. Write like you're excited to show someone how something works.
- Use "we" and "you" constantly. The reader is right there with you.
- Mix short punchy sentences with medium-length explanatory ones. Never academic.
- Use fragments to land a point: "Pretty neat." "Simple." "That's it."
- Honest hedges when appropriate: "I have no idea if this is still the best way" beats false confidence.

### Hooks and Openings

- Lead with curiosity, not a thesis. Questions work: "Have you ever wondered how a touch screen knows you're touching it?"
- Or lead with the concrete thing itself: an example, a visual description, a code block, a surprising fact.
- Never open with a grand claim or landscape summary.

### Show-Then-Explain Pattern

- Show the thing first (example, diagram description, code, output, visual), then explain how it works and why.
- Build concepts in layers — each section adds one piece, progressive revelation.
- Let visuals and examples do the heavy lifting. The prose connects the dots between them.

### Technical Depth

- Make genuinely complex topics accessible without dumbing them down.
- Use analogies grounded in everyday experience.
- Show the actual math, code, or mechanism when it matters — but frame it so it doesn't scare people off.
- Acknowledge messiness: "It feels super convoluted and is a tad slow but the performance isn't that bad."
- Never assume expertise. Give people permission: "You don't need to be technical to follow this."

### Humor and Personality

- Light and self-aware, never try-hard. A dry aside, a brief self-deprecating remark.
- Occasional casual interjections: "Well...", "Okay, so...", "Luckily for us..."
- Give the reader permission to skip: "Feel free to skip this if you don't care."
- Emoji sparingly and only in blog/tutorial contexts, not in essays or formal pieces.

### Formatting Habits

- Short paragraphs: 1-3 sentences max.
- Clear section headings, often as questions ("How does it work?") or short noun phrases.
- Inline code formatting for technical terms.
- Bullet lists for options, steps, or comparisons.
- Code blocks with brief setup context, never dropped in without framing.

## Core Rules

1. Show first, explain after. Example/visual/code block, then the why.
2. Every sentence earns its place. If it restates the previous one, cut it.
3. Use specific numbers when available and sourced.
4. Never invent biographical facts, company metrics, or customer evidence.
5. Write like you're explaining to a curious person, not lecturing a student.

## Voice Capture Workflow

If the user wants a voice different from the default, collect one or more of:

- published articles
- newsletters
- X / LinkedIn posts
- docs or memos
- a short style guide

Then extract:

- sentence length and rhythm
- whether the voice is formal, conversational, or sharp
- favored rhetorical devices such as parentheses, lists, fragments, or questions
- tolerance for humor, opinion, and contrarian framing
- formatting habits such as headers, bullets, code blocks, and pull quotes

## Banned Patterns

Delete and rewrite any of these:

- generic openings: "In today's rapidly evolving landscape", "As we all know..."
- filler transitions: "Moreover", "Furthermore", "It's worth noting that"
- hype words: "game-changer", "cutting-edge", "revolutionary", "groundbreaking"
- vague claims without evidence or examples
- prescriptive lecturing: "You should always..." (prefer "One thing that helps is...")
- biography or credibility claims not backed by provided context
- throat-clearing first paragraphs that delay the actual content

## Writing Process

1. Clarify the audience and purpose.
2. Build a skeletal outline — one concept per section, ordered for progressive revelation.
3. Start each section with the thing itself: the example, the visual, the code, the question.
4. Explain after showing. Connect the example to the concept.
5. Expand only where the next sentence adds new information.
6. Remove anything that sounds templated, self-congratulatory, or like a textbook.

## Structure Guidance

### Technical Explainers (default for this voice)

- open with a curiosity hook or a concrete "here's the thing"
- one concept per section, building on the last
- describe or reference visuals/diagrams — let them carry complexity
- use code or terminal examples where they clarify
- end with what the reader now understands, not a fluffy summary

### Technical Guides / Tutorials

- open with what the reader will build or accomplish
- use code or terminal examples in every major section
- be honest about rough edges and alternatives
- end with concrete takeaways or next steps

### Essays / Opinion Pieces

- start with tension, contradiction, or a sharp observation
- keep one argument thread per section
- use examples that earn the opinion

### Newsletters

- keep the first screen strong
- mix insight with updates, not diary filler
- use clear section labels and easy skim structure

## Quality Gate

Before delivering:

- verify factual claims against provided sources
- remove filler and corporate language
- confirm the voice matches the default (curious, warm, honest) or supplied examples
- ensure every section adds new information
- check that the show-then-explain pattern is followed, not explain-then-show
- check formatting for the intended platform
